# Dispatch Profiles

`profiles/` catalogs **dispatch profiles** — declarations that build a masked,
partial-projection config home for a single headless dispatch. A profile
selects a harness, a model (portable role or concrete), an L2 role fragment,
and an exposed subset of skills/agents/triggers. The home is generated by
`tools/profile/build-home.py` from a single repo source via symlink
partial-projection — there is no content fork.

## The three-layer instruction model

See `spec/dispatch-profiles/prd.md` §2 for the full rationale. Summary:

| Layer | Content | Exposure |
|---|---|---|
| **L0 core contract** | `core/` (CORE/CONVENTIONS/OPERATIONS/WORKFLOW) + guard hooks (artifact-guard, git-state-guard, spec-skill-gate, builtin-memory-guard) + git discipline | Always included, unmaskable, for every profile |
| **L1 orchestration** | Dispatch/harvest/routing/fleet control — the main session's CLAUDE.md body | Main only — masked out of every dispatch profile |
| **L2 role fragment** | Case-specific specialization (`profiles/fragments/<name>.md`) — experiment-domain rules, studio-style role contracts, doc-tone contracts, etc. | That profile only — masked out of main |

Masking hides *body*, not *existence* — main can always route via this
README's name + one-line description, same contract as the skill catalog.
Masking is not inaccessibility — fragments are ordinary repo files main can
Read on demand; only the automatic-injection scope differs per profile.

## `profiles/` is a scoped exception to the portable-boundary rule

`profiles/` sits at top level like `roles/`, but unlike `roles/` it is a
**harness-scoped declaration directory** — every `profiles/<name>.yaml` binds
a `harness:` field, and concrete model names are permitted *inside that
binding*. This is the **intended exception** to the canonical rule in
`core/ADAPTATION.md` ("Concrete model names belong in adapter documents or
generated native files") — the rule still holds for `roles/` (portable,
harness-neutral), but a dispatch profile declaration is itself a
harness-bound adapter-facing artifact, so a concrete `model:` field there is
the rule's exception surface, not a violation of it. Prefer `model_role`
(portable) unless a concrete model is required (e.g. an opencode
glm/deepseek provider model) — see the individual `<name>.yaml` files.

## Term disambiguation — not the same "profile"

The **dispatch profile** described here (`profiles/<name>.yaml`, the
`--profile <name>` dispatch-wrapper flag, and the `profile=<name>` jobs.log
`pipe` field — together a masked per-dispatch config home) is a **different
concept** from the family's existing **mem-profile** (`mem profile <stem>`,
DB `type=profile` records, user personalization; see `core/MEMORY.md` §7.6).
Both are called "profile" in their own context — do not conflate them in
docs, grep, or recall.
